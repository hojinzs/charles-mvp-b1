# 네이버 키워드 광고 + 검색 순위 모니터링 POC 스펙 시트

## 1. 프로젝트 목적 (Purpose)

본 프로젝트는 **네이버 키워드 광고 API와 실제 검색 화면 관측을 결합하여**,
다음 질문에 대해 기술적으로 명확한 답을 얻는 것을 목적으로 한다.

- 특정 키워드에 대해
  **내가 집행 중인 광고 또는 특정 타겟 URL이**
  - 실제 네이버 검색 화면에 노출되는지
  - 노출된다면 대략 어느 위치에 보이는지
  **자동으로 수집·판별할 수 있는가**

이를 통해 단순한 "순위 크롤링"이 아니라,
**광고 집행 상태 + 검색 노출 상태를 함께 해석할 수 있는 시스템이 가능한지**를 검증한다.

본 POC는 다음을 증명하기 위한 것이다.

- 네이버 광고 API만으로는 알 수 없는 "실제 화면 기준 노출 상태"를
  기술적으로 보완할 수 있는가
- 이 구조가 향후 대량 키워드 모니터링 서비스로 확장 가능한가
- 비개발자(마케터/운영자)가 직접 사용하고 판단할 수 있는 UX를 제공할 수 있는가

---

## 2. 문제 정의 (Problem Statement)

네이버 키워드 광고 운영 및 검색 마케팅에서 다음과 같은 한계가 존재한다.

- 광고 API는 평균 광고 노출 위치만 제공하며,
  실제 검색 화면에서의 노출 여부나 상대적 위치를 제공하지 않는다.
- 실제 검색 화면 기준 순위는 공식 API로 제공되지 않는다.
- 광고 집행 중임에도 검색 화면에서 보이지 않는 경우에 대한
  직관적인 판단 수단이 부족하다.

본 프로젝트는 위 한계를 **광고 API + 화면 관측 기반 크롤링의 조합**으로
현실적으로 해결할 수 있는지 검증한다.

---

## 3. 범위 정의 (Scope)

### 3.1 포함 범위 (In Scope)

- 네이버 통합검색 기준
- 키워드 단위 분석
- 다음 두 대상에 대한 노출 여부 및 위치 판별
  - 내가 집행 중인 광고 (광고 영역)
  - 특정 타겟 URL 또는 도메인 (자연 검색 영역)
- PC 검색 기준
- 수동 실행 기반 POC (스케줄 자동화는 제외)

---

### 3.2 제외 범위 (Out of Scope)

- 실시간 100% 정확한 광고 순위 보장
- 사용자별 개인화 검색 결과 완전 재현
- 네이버 공식 API만을 사용한 순위 산출
- 수만 개 키워드의 고빈도 실시간 운영

본 POC는 **기술 가능성 검증 단계**로 한정한다.

---

## 4. 기술 스택

### 4.1 플랫폼 및 언어

- Node.js
- Electron
- React
- SQLite

---

### 4.2 주요 구성 요소

- 네이버 키워드 광고 API 연동 모듈
- 검색 화면 관측 기반 크롤링 모듈
- 작업 큐 및 실행 제어 모듈
- 로컬 데이터 저장소
- 비개발자 검증용 UI

---

## 5. 시스템 아키텍처 개요

본 POC는 서버 분리 없이 단일 Electron 애플리케이션으로 구성된다.

- React UI는 사용자 입력 및 결과 확인을 담당한다.
- Electron Main Process는 다음 역할을 수행한다.
  - 광고 API 호출
  - 작업 큐 관리
  - 검색 화면 관측 및 순위 판별
  - 결과 저장

이 구조는 향후 서버 기반 워커 아키텍처로 자연스럽게 이전 가능하도록 설계한다.

---

## 6. 주요 기능 상세

### 6.1 키워드 입력 및 관리

- 사용자는 여러 키워드를 입력할 수 있다.
- 각 키워드에 대해 타겟 도메인 또는 랜딩 URL을 지정할 수 있다.
- 키워드는 추적 대상 여부를 개별적으로 설정할 수 있다.

---

### 6.2 네이버 키워드 광고 API 활용

#### 사용 목적

- 내가 실제로 집행 중인 키워드인지 확인
- 키워드별 광고 시장 규모 및 경쟁도 파악
- 평균 광고 노출 위치 정보를 확보하여
  검색 화면 관측 결과 해석에 활용

#### 활용 데이터

- 월간 검색량 (PC / Mobile)
- 평균 클릭 단가
- 경쟁도
- 평균 광고 노출 위치 지표

---

### 6.3 검색 화면 관측 및 순위 판별

- 실제 네이버 검색 화면을 렌더링하여 관측한다.
- 광고 영역과 자연 검색 영역을 구분한다.
- 각 영역에서 다음을 판별한다.
  - 내 광고 랜딩 URL의 존재 여부
  - 타겟 도메인의 최초 등장 위치

순위는 "현재 검색 화면 기준 참고 값"으로 기록한다.

---

### 6.4 작업 큐 및 실행 제어

- 키워드 단위 작업 큐를 사용한다.
- 동시 실행 수 및 요청 간 간격을 제한한다.
- 개별 작업 실패 시 전체 실행은 중단되지 않는다.
- 실행 및 일시정지가 가능해야 한다.

---

## 7. 데이터 저장

로컬 SQLite 데이터베이스를 사용한다.

### 저장 대상

- 키워드 기본 정보
- 광고 메트릭 정보
- 검색 결과 기반 순위 데이터
- 작업 실행 로그

데이터 구조는 향후 RDB 또는 분석용 DB로 이전 가능하도록 단순하게 유지한다.

---

## 8. UI 요구사항

본 POC의 UI는 **비개발자 검증**을 주요 기준으로 한다.

### 필수 기능

- 키워드 입력 및 관리
- 광고 데이터 조회 및 표시
- 실행 상태 및 진행률 표시
- 키워드별 결과 확인
- 결과 데이터 확인 및 내보내기

디자인 완성도보다 **이해 가능성과 신뢰감**을 우선한다.

---

## 9. 결과 해석 원칙

본 POC는 다음 원칙을 따른다.

- 광고 API의 평균 광고 노출 위치와
  검색 화면 관측 결과는 동일하지 않을 수 있다.
- 검색 화면 기준 순위는 참고용 지표로 제공한다.
- UI 및 리포트에서는 오해를 방지하는 표현을 사용한다.

---

## 10. 성공 기준 (POC Exit Criteria)

다음 조건을 만족하면 POC는 성공으로 판단한다.

1. 비개발자가 직접 키워드를 입력하고 실행할 수 있다.
2. 광고가 노출된 경우와 노출되지 않은 경우를 구분할 수 있다.
3. 검색 화면 기준 노출 위치를 합리적으로 설명할 수 있다.
4. 큐 기반 실행 구조가 정상적으로 동작한다.
5. 향후 대규모 확장이 가능하다는 기술적 판단이 가능하다.

---

## 11. 향후 확장 방향

- 서버 기반 워커 아키텍처로 이전
- 분산 큐 도입
- 대량 키워드 스케줄링
- 모바일 검색 지원
- 광고·검색 데이터를 결합한 자동 진단 로직

---

## 요약

본 POC는
**"네이버 광고 API의 한계를 실제 검색 화면 관측으로 보완하여
광고 집행 상태와 검색 노출 상태를 함께 해석할 수 있는 시스템이 가능한가"를
가장 작은 단위에서 검증하기 위한 프로젝트**다.
